# Phase 10 — Validation Strategy

> Validation Architecture generated by gsd-phase-researcher to guide the gsd-executor.

## The Nyquist Loop

This phase implements a backend `target_groups` domain for tracking target FB groups and scraped posts, along with a `TargetGroups.tsx` frontend page. The execution must be validated at the data layer, background layer, and presentation layer.

## Verification Checklist

### 1. Data Layer & API
- [ ] **Create Target Group**: API `POST /api/v1/target_groups` successfully inserts a target group.
- [ ] **Unique Posts**: Duplicate `ScrapedPost` with the same `original_url` must be silently skipped/ignored upon insert.
- [ ] **Approval Status Update**: API `PATCH /api/v1/target_groups/posts/{id}/status` correctly shifts status from `PENDING` to `APPROVED` or `REJECTED`.

### 2. Background Scraper Layer
- [ ] **Scheduler Tick**: `facebook_scheduler_tick` task registers correctly in Celery and spawns `scrape_facebook_group` for active groups.
- [ ] **Playwright Execution**: Scraper can successfully fetch a Facebook group page and parse at least one post, extracting its URL, author, and content snippet.

### 3. Frontend UI Layer
- [ ] **Group Management Tab/Card**: Displays configured target groups and allows adding/deleting them.
- [ ] **Scraped Posts Table**: Table view correctly lists `PENDING` posts with "Approve" and "Reject" actions.
- [ ] **Status Change Reaction**: Clicking "Approve" acts properly and re-fetches or modifies the status on the UI table.
